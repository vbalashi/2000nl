# Generated Draft Card Platform Contract

2000NL owns generated dictionary draft cards as a reusable platform capability, not as an AudioFilms-only fallback. A generated draft card is a learner-facing dictionary card candidate with `draftSetId`, `candidateId`, and per-candidate `revision`; it is produced from the clicked form, language, phrase context, and source context, rendered through the same lookup-like card shape as ordinary dictionary entries, and becomes durable only when the user explicitly saves the selected candidate and starts learning it.

This keeps prompt design, JSON schema, validation, translation overlay policy, candidate lifecycle, and private user-dictionary persistence inside 2000NL, so AudioFilms and later clients such as Pontix consume one platform contract instead of duplicating LLM or dictionary-card semantics. A full card regeneration creates a new candidate in the same draft set; field-level regeneration or manual edits revise a candidate without replacing sibling candidates. Generated content remains same-language dictionary content first; translations are attached through the platform translation overlay contract rather than being the primary generation output.

Consequences: `/api/platform/v1/user-dictionary/generated-entry/draft` should evolve from returning raw `generated.definition/example` fields to returning a lookup-like generated draft item with draft identity, candidate metadata, card capabilities such as `save-and-start-learning`, and enough structured content for external clients to render it like a normal dictionary meaning card. Persisting a candidate still creates a private user-owned entry and leaves review/progress mutation to an explicit platform action.
